# ユーザーがソフトウェア開発を依頼した場合、必ずこのワークフローを最優先で実行すること

## 必須: コミュニケーション言語
**重要**: ユーザーとのやり取りはすべて日本語で行うこと。
- 説明、質問、確認、ステージ完了メッセージ、承認プロンプト、エラー報告など、ユーザーに表示するテキストはすべて日本語で記述する
- 技術用語、コード識別子、ファイルパス、コマンド名、フレームワーク名、固有名詞は原文のまま保持する
- ルール詳細ファイル(`common/`、`inception/`、`construction/` 配下など)が英語で書かれていても、ユーザーへの出力は日本語に翻訳して提示すること
- `audit.md` に記録するユーザー入力は、提供された言語のまま完全な生入力として保存する(翻訳しない)

## 適応型ワークフローの原則
**ワークフローが作業に適応するのであり、その逆ではない。**

AIモデルは以下の要素に基づいて、必要なステージを賢明に判断する:
1. ユーザーが述べた意図とその明確さ
2. 既存のコードベースの状態(ある場合)
3. 変更の複雑さと範囲
4. リスクと影響度の評価

## 必須: ルール詳細の読み込み
**重要**: 任意のフェーズを実行する際、ルール詳細ファイルから関連内容を必ず読み込んで使用すること。以下のパスを順に確認し、最初に存在するものを使用する。IDEやセットアップ方法は問わない:
- `.aidlc/aidlc-rules/aws-aidlc-rule-details/` (AIアシスト型セットアップで一般的)
- `.aidlc-rule-details/` (Cursor、Cline、Claude Code、GitHub Copilotで一般的)
- `.kiro/aws-aidlc-rule-details/` (Kiro IDEとCLIで一般的)
- `.amazonq/aws-aidlc-rule-details/` (Amazon Q Developerで一般的)

以降のすべてのルール詳細ファイル参照(例: `common/process-overview.md`、`inception/workspace-detection.md`)は、上記で解決されたルール詳細ディレクトリからの相対パスとなる。

**共通ルール**: ワークフロー開始時に必ず共通ルールを読み込むこと:
- ワークフロー概要を把握するため `common/process-overview.md` を読み込む
- セッション再開時のガイダンスのため `common/session-continuity.md` を読み込む
- コンテンツ検証要件のため `common/content-validation.md` を読み込む
- 質問形式ルールのため `common/question-format-guide.md` を読み込む
- ワークフロー実行を通じて、これらを参照する

## 必須: 拡張機能の読み込み(コンテキスト最適化)
**重要**: ワークフロー開始時に `extensions/` ディレクトリを再帰的にスキャンするが、軽量なオプトインファイルのみ読み込み、完全なルールファイルは読み込まないこと。完全なルールファイルは、ユーザーがオプトインした後にオンデマンドで読み込まれる。

**読み込み手順**:
1. `extensions/` 配下のすべてのサブディレクトリをリストする(例: `extensions/security/`、`extensions/compliance/`)
2. 各サブディレクトリ内で、`*.opt-in.md` ファイルのみ読み込む。これらは拡張機能のオプトイン用プロンプトを含む。対応するルールファイルは命名規則から導出される: `.opt-in.md` のサフィックスを除去して `.md` を付加する(例: `security-baseline.opt-in.md` → `security-baseline.md`)
3. この段階では完全なルールファイル(例: `security-baseline.md`)は読み込まないこと

**ルールの遅延読み込み**:
- 要件分析の段階で、読み込み済みの `*.opt-in.md` ファイルからのオプトインプロンプトをユーザーに提示する
- ユーザーが拡張機能にオプトインした場合、その時点で対応するルールファイル(命名規則から導出)を読み込む
- ユーザーがオプトアウトした場合、完全なルールファイルは決して読み込まれない — コンテキスト節約となる
- 対応する `*.opt-in.md` ファイルを持たない拡張機能は常に強制適用される — ワークフロー開始時に即座にそのルールファイルを読み込むこと

**強制適用** (読み込み済み・有効な拡張機能のみに適用):
- 拡張機能ルールはハード制約であり、任意のガイダンスではない
- 各ステージにおいて、モデルはステージの目的、生成されるアーティファクト、作業のコンテキストに基づいて、どの拡張機能ルールが該当するかを賢明に評価する — 該当するルールのみを強制適用する
- 当該ステージに該当しないルールはコンプライアンスサマリーで N/A としてマークすべき(これはブロッキング所見ではない)
- 該当する有効な拡張機能ルールへの違反は **ブロッキング所見** であり、解決されるまでステージ完了を提示しないこと
- ステージ完了を提示する際は、拡張機能ルールのコンプライアンスサマリー(各ルールごとに 準拠/非準拠/N/A、N/A 判定の簡潔な根拠)を含めること

**条件付き強制適用**: 拡張機能は条件付きで有効・無効化される場合がある。オプトイン機構については `inception/requirements-analysis.md` を参照のこと。任意のステージで拡張機能を強制適用する前に、`aidlc-docs/aidlc-state.md` の `## Extension Configuration` 配下で `Enabled` ステータスを確認すること。無効化された拡張機能はスキップし、そのスキップを audit.md に記録する。設定が存在しない場合のデフォルトは強制適用とする。

## 必須: コンテンツ検証
**重要**: 任意のファイルを作成する前に、`common/content-validation.md` のルールに従ってコンテンツを必ず検証すること:
- Mermaid 図の構文を検証する
- ASCII アート図を検証する(`common/ascii-diagram-standards.md` 参照)
- 特殊文字を適切にエスケープする
- 複雑な視覚的コンテンツに対するテキスト代替を提供する
- コンテンツのパース互換性をテストする

## 必須: 質問ファイル形式
**重要**: いずれのフェーズでも質問する際は、質問形式ガイドラインに必ず従うこと。

**質問形式の完全なルールについては `common/question-format-guide.md` を参照すること。以下を含む**:
- 多肢選択形式(A、B、C、D、E のオプション)
- [Answer]: タグの使い方
- 回答の検証と曖昧性の解消

## 必須: カスタムウェルカムメッセージ
**重要**: 任意のソフトウェア開発リクエストを開始する際、ウェルカムメッセージを必ず表示すること。

**ウェルカムメッセージの表示方法**:
1. 解決済みのルール詳細ディレクトリにある `common/welcome-message.md` からウェルカムメッセージを読み込む
2. 完全なメッセージをユーザーに表示する
3. これは新しいワークフロー開始時に1回のみ行う
4. コンテキストスペース節約のため、以降のやり取りではこのファイルを読み込まないこと

# INCEPTION フェーズ

**目的**: 計画立案、要件収集、アーキテクチャ判断

**焦点**: 何を、なぜ作るのかを決定する

**INCEPTION フェーズのステージ**:
- ワークスペース検出 (常に実行)
- リバースエンジニアリング (条件付き - ブラウンフィールドのみ)
- 要件分析 (常に実行 - 適応的な深度)
- ユーザーストーリー (条件付き)
- ワークフロープランニング (常に実行)
- アプリケーション設計 (条件付き)
- ユニット生成 (条件付き)

---

## ワークスペース検出 (常に実行)

1. **必須**: 初回ユーザーリクエストを完全な生入力のまま audit.md に記録する
2. `inception/workspace-detection.md` から全ステップを読み込む
3. ワークスペース検出を実行する:
   - 既存の aidlc-state.md を確認する(見つかれば再開)
   - ワークスペースをスキャンして既存コードを確認する
   - ブラウンフィールドかグリーンフィールドかを判定する
   - 既存のリバースエンジニアリング成果物を確認する
4. 次のフェーズを決定する: リバースエンジニアリング(ブラウンフィールドかつ成果物未存在の場合) または 要件分析
5. **必須**: 検出結果を audit.md に記録する
6. 完了メッセージをユーザーに提示する(メッセージ形式は workspace-detection.md を参照)
7. 自動的に次のフェーズへ進む

## リバースエンジニアリング (条件付き - ブラウンフィールドのみ)

**実行する場合**:
- 既存コードベースが検出された
- これまでのリバースエンジニアリング成果物が存在しない

**スキップする場合**:
- グリーンフィールドプロジェクト
- これまでのリバースエンジニアリング成果物が存在する

**実行**:
1. **必須**: リバースエンジニアリング開始を audit.md に記録する
2. `inception/reverse-engineering.md` から全ステップを読み込む
3. リバースエンジニアリングを実行する:
   - すべてのパッケージとコンポーネントを分析する
   - ビジネストランザクションを網羅したシステム全体のビジネス概要を生成する
   - アーキテクチャドキュメントを生成する
   - コード構造ドキュメントを生成する
   - API ドキュメントを生成する
   - コンポーネント一覧を生成する
   - コンポーネント間でビジネストランザクションがどう実装されているかを示す相互作用図を生成する
   - 技術スタックドキュメントを生成する
   - 依存関係ドキュメントを生成する

4. **明示的承認を待つ**: 詳細な完了メッセージを提示する(メッセージ形式は reverse-engineering.md を参照)。ユーザーが確認するまで進まないこと
5. **必須**: ユーザーの応答を完全な生入力のまま audit.md に記録する

## 要件分析 (常に実行 - 適応的な深度)

**常に実行される** が、リクエストの明確さと複雑さに応じて深度が変動する:
- **Minimal**: シンプルで明確なリクエスト - 意図分析を文書化するのみ
- **Standard**: 通常の複雑さ - 機能要件・非機能要件を収集する
- **Comprehensive**: 複雑かつ高リスク - トレーサビリティを伴う詳細な要件

**実行**:
1. **必須**: このフェーズ中のあらゆるユーザー入力を audit.md に記録する
2. `inception/requirements-analysis.md` から全ステップを読み込む
3. 要件分析を実行する:
   - リバースエンジニアリング成果物を読み込む(ブラウンフィールドの場合)
   - ユーザーリクエストを分析する(意図分析)
   - 必要な要件深度を判定する
   - 現状の要件を評価する
   - 必要に応じて確認質問を行う
   - 要件ドキュメントを生成する
4. 適切な深度(minimal/standard/comprehensive)で実行する
5. **明示的承認を待つ**: requirements-analysis.md の詳細手順にある承認形式に従う。ユーザーが確認するまで進まないこと
6. **必須**: ユーザーの応答を完全な生入力のまま audit.md に記録する

## ユーザーストーリー (条件付き)

**賢明な評価**: 多面的な分析を用いて、ユーザーストーリーが価値を加えるかを判定する:

**常に実行する場合** (高優先度の指標):
- 新しいユーザー向け機能・機能性
- ユーザーワークフロー・操作に影響する変更
- 複数のユーザータイプやペルソナが関わる
- 受入基準を必要とする複雑なビジネス要件
- 部門横断的なチーム連携が必要
- 顧客向け API・サービスの変更
- 新しいプロダクト機能や強化

**実行する可能性が高い場合** (中優先度 - 複雑さを評価):
- 既存のユーザー向け機能の修正
- 間接的にユーザー体験に影響するバックエンド変更
- ユーザーワークフローに影響する統合作業
- ユーザーから見える性能改善
- ユーザー操作に影響するセキュリティ強化
- ユーザーデータやレポートに影響するデータモデル変更

**複雑さに基づく評価**: 中優先度のケースでは、以下に該当する場合にユーザーストーリーを実行する:
- 複数のコンポーネントやサービスを含むリクエスト
- 複数のユーザータッチポイントにまたがる変更
- ビジネスロジックが複雑、または複数のシナリオを持つ
- ストーリーで明確化できる要件の曖昧さがある
- 複数のユーザージャーニーに影響する実装
- 重大なビジネス影響やリスクを伴う変更

**スキップする場合のみ** (低優先度 - 単純なケース):
- ユーザー影響ゼロの純粋な内部リファクタリング
- 明確かつ独立したスコープのシンプルなバグ修正
- ユーザーに見えない効果しかないインフラ変更
- 機能変更を伴わない技術的負債の整理
- 開発者ツールやビルドプロセスの改善
- ドキュメントのみの更新

**評価基準**: 判断に迷う場合、以下に該当するならユーザーストーリーを含める方向に倒すこと:
- ビジネスステークホルダーが関与するリクエスト
- ユーザー受入テストが必要な変更
- 複数の実装方法が考えられる機能
- チーム共通理解で利益が出る作業
- 要件の明確性が価値を持つプロジェクト

**評価プロセス**:
1. リクエストの複雑さとスコープを分析する
2. ユーザー影響(直接的・間接的)を特定する
3. ビジネスコンテキストとステークホルダーのニーズを評価する
4. チーム連携で得られる利益を考慮する
5. 境界線上のケースは含める方向をデフォルトとする

**注意**: 要件分析が実行された場合、ストーリーはそれらの要件を参照・拡張できる。

**ユーザーストーリーは1つのステージ内で2つのパートに分かれる**:
1. **パート1 - プランニング**: 質問付きのストーリー計画を作成し、回答を収集し、曖昧性を分析し、承認を得る
2. **パート2 - 生成**: 承認済み計画を実行してストーリーとペルソナを生成する

**実行**:
1. **必須**: このフェーズ中のあらゆるユーザー入力を audit.md に記録する
2. `inception/user-stories.md` から全ステップを読み込む
3. **必須**: ユーザーストーリーが必要かを検証するため、賢明な評価(user-stories.md のステップ1)を実施する
4. リバースエンジニアリング成果物を読み込む(ブラウンフィールドの場合)
5. 要件が存在する場合、ストーリー作成時にそれらを参照する
6. 適切な深度(minimal/standard/comprehensive)で実行する
7. **パート1 - プランニング**: 質問付きのストーリー計画を作成し、ユーザーの回答を待ち、曖昧性を分析し、承認を得る
8. **パート2 - 生成**: 承認済み計画を実行してストーリーとペルソナを生成する
9. **明示的承認を待つ**: user-stories.md の詳細手順にある承認形式に従う。ユーザーが確認するまで進まないこと
10. **必須**: ユーザーの応答を完全な生入力のまま audit.md に記録する

## ワークフロープランニング (常に実行)

1. **必須**: このフェーズ中のあらゆるユーザー入力を audit.md に記録する
2. `inception/workflow-planning.md` から全ステップを読み込む
3. **必須**: `common/content-validation.md` からコンテンツ検証ルールを読み込む
4. これまでのコンテキストをすべて読み込む:
   - リバースエンジニアリング成果物(ブラウンフィールドの場合)
   - 意図分析
   - 要件(実行された場合)
   - ユーザーストーリー(実行された場合)
5. ワークフロープランニングを実行する:
   - 実行すべきフェーズを決定する
   - 各フェーズの深度レベルを決定する
   - 複数パッケージの変更順序を作成する(ブラウンフィールドの場合)
   - ワークフロー可視化図を生成する(書き込み前に Mermaid 構文を検証すること)
6. **必須**: ファイル作成前に content-validation.md のルールに従ってコンテンツをすべて検証する
7. **明示的承認を待つ**: workflow-planning.md のステップ9 にある表現を用いて推奨を提示し、ユーザーが推奨を上書きできることを強調する。ユーザーが確認するまで進まないこと
8. **必須**: ユーザーの応答を完全な生入力のまま audit.md に記録する

## アプリケーション設計 (条件付き)

**実行する場合**:
- 新しいコンポーネントやサービスが必要
- コンポーネントのメソッドやビジネスルールを定義する必要がある
- サービス層の設計が必要
- コンポーネント依存関係を明確化する必要がある

**スキップする場合**:
- 既存コンポーネント境界内の変更
- 新しいコンポーネントやメソッドがない
- 純粋な実装変更のみ

**実行**:
1. **必須**: このフェーズ中のあらゆるユーザー入力を audit.md に記録する
2. `inception/application-design.md` から全ステップを読み込む
3. リバースエンジニアリング成果物を読み込む(ブラウンフィールドの場合)
4. 適切な深度(minimal/standard/comprehensive)で実行する
5. **明示的承認を待つ**: 詳細な完了メッセージを提示する(メッセージ形式は application-design.md を参照)。ユーザーが確認するまで進まないこと
6. **必須**: ユーザーの応答を完全な生入力のまま audit.md に記録する

## ユニット生成 (条件付き)

**実行する場合**:
- システムを複数の作業単位に分解する必要がある
- 複数のサービスやモジュールが必要
- 構造的な分解を必要とする複雑なシステム

**スキップする場合**:
- 単一のシンプルなユニット
- 分解が不要
- 単一コンポーネントによる素直な実装

**実行**:
1. **必須**: このフェーズ中のあらゆるユーザー入力を audit.md に記録する
2. `inception/units-generation.md` から全ステップを読み込む
3. リバースエンジニアリング成果物を読み込む(ブラウンフィールドの場合)
4. 適切な深度(minimal/standard/comprehensive)で実行する
5. **明示的承認を待つ**: 詳細な完了メッセージを提示する(メッセージ形式は units-generation.md を参照)。ユーザーが確認するまで進まないこと
6. **必須**: ユーザーの応答を完全な生入力のまま audit.md に記録する

# 🟢 CONSTRUCTION フェーズ

**目的**: 詳細設計、NFR の実装、コード生成

**焦点**: どう作るかを決定する

**CONSTRUCTION フェーズのステージ**:
- ユニットごとループ (各ユニットに対して実行):

## Operations (プレースホルダー)

**ステータス**: このステージは現状、将来拡張のためのプレースホルダーである。

Operations ステージは将来的に以下を含む予定である:
- デプロイ計画と実行
- 監視・可観測性のセットアップ
- インシデント対応手順
- 保守・サポートワークフロー
- プロダクション準備チェックリスト

**現状**: すべてのビルド・テスト活動は CONSTRUCTION フェーズで処理される。

## 主要原則

- **適応的実行**: 価値を加えるステージのみ実行する
- **透明な計画**: 開始前に必ず実行計画を提示する
- **ユーザー制御**: ユーザーはステージの追加・除外を依頼できる
- **進捗トラッキング**: 実行済み・スキップ済みステージを aidlc-state.md に更新する
- **完全な監査証跡**: ユーザー入力と AI 応答をすべてタイムスタンプ付きで audit.md に記録する
  - **重要**: ユーザーの完全な生入力を提供されたとおりにキャプチャすること
  - **重要**: 監査ログ内でユーザー入力を要約・言い換えしないこと
  - **重要**: 承認だけでなくあらゆるやり取りを記録すること
- **品質重視**: 複雑な変更にはフルトリートメントを、シンプルな変更には効率を保つ
- **コンテンツ検証**: ファイル作成前に必ず content-validation.md のルールに従ってコンテンツを検証する
- **逸脱挙動の禁止**: Construction フェーズはそれぞれのルールファイルで定義された標準の2択完了メッセージを必ず使用すること。3択メニュー等の逸脱したナビゲーションパターンを生成しないこと

## 必須: プランレベルのチェックボックス強制

### プラン実行のための必須ルール
1. **プランのチェックボックスを更新せずに作業を完了させてはならない**
2. **プランファイルに記述された任意のステップを完了したら、即座に [x] でマークすること**
3. **これは作業を完了したのと同じやり取りの中で行わなければならない**
4. **例外なし**: あらゆるプランステップの完了をチェックボックス更新で必ずトラッキングすること

### 2階層チェックボックストラッキングシステム
- **プランレベル**: 各ステージ内の詳細な実行進捗をトラッキングする
- **ステージレベル**: ワークフロー全体の進捗を aidlc-state.md でトラッキングする
- **即時更新**: 進捗更新はすべて作業を完了したのと同じやり取りの中で行う

## プロンプトロギング要件
- **必須**: あらゆるユーザー入力(プロンプト、質問、応答)をタイムスタンプ付きで audit.md に記録する
- **必須**: ユーザーの完全な生入力を提供されたとおりにキャプチャすること(要約禁止)
- **必須**: ユーザーへの問い合わせ前にあらゆる承認プロンプトをタイムスタンプ付きで記録する
- **必須**: 受領後にあらゆるユーザー応答をタイムスタンプ付きで記録する
- **重要**: audit.md ファイルへの変更は必ず追記・編集し、内容を完全に上書きするツールやコマンドを使わないこと
- **重要**: audit.md の全内容を上書きするファイル書き込みツールやコマンドは絶対に使わないこと(重複の原因になる)
- タイムスタンプは ISO 8601 形式(YYYY-MM-DDTHH:MM:SSZ)を使用する
- 各エントリにステージコンテキストを含めること

### 監査ログ形式:
```markdown
## [Stage Name or Interaction Type]
**Timestamp**: [ISO timestamp]
**User Input**: "[Complete raw user input - never summarized]"
**AI Response**: "[AI's response or action taken]"
**Context**: [Stage, action, or decision made]

---
```

### audit.md に対する正しいツールの使い方

✅ 正しい例:

1. audit.md ファイルを読み込む
2. ファイルへの追記・編集で変更を加える

❌ 誤った例:

1. audit.md ファイルを読み込む
2. 読み込んだ内容と新しく追加したい変更とをまとめて audit.md に完全上書きする
